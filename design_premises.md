## 1. ユースケース定義（条件の具体化）

- **デバイス**：「800GbE Ethernet controller」＝ 最上位の **RDMA 対応 NIC/DPU**。PCIe Gen6 x16 をほぼ使い切る帯域整合の高級デバイス（汎用 NIC ではない）。NIC は DMA マスタとして IOVA 付き書き込みを発行し、IOMMU がそれを物理アドレスへ変換する。
- **用途**：AI/ML 分散学習（勾配/パラメータ/活性の交換, RoCEv2）、NVMe-oF/ストレージ、チェックポイント、データセット取り込み。共通項は「マシン間の最高速バルク転送」。Tenstorrent 文脈ではアクセラレータのスケールアウト・ファブリック。
- **「連続4kB→連続IOVA」とは**：1つの大きな論理転送が1つの連続バッファに着地するパターン。代表は **大メッセージの RDMA WRITE を登録済み連続バッファ(MR)へ順次書き込む**。小パケットRX（多数の小バッファに散る）とは別物＝帯域律速のバルクDMA。

## 2. 固定条件と導出数値

| 項目 | 値 |
|---|---|
| 実効ワイヤレート（律速） | 800Gbps = **100 GB/s**（PCIe 128GB/s は非律速） |
| ページ | 4 kB |
| 変換レート | 100e9 / 4096 = **24.41 M/s** |
| 到着間隔 | **40.96 ns ≈ 16.4 cycles** |
| IOMMU クロック | 400 MHz（2.5 ns/cycle） |
| メモリ | AXI, **100 ns** レイテンシ |
| no-cache 3段ウォーク | 3×100 = **300 ns** |
| 必要並列ウォーカー(no-cache, Little) | 300/40.96 ≈ **8** |
| 必要バッファ(no-cache, Little) | ≈ **8** |
| 64B キャッシュライン | = **8 PTE**（PTE=8B）→ leaf コアレッシング係数 8 |
| nested(Sv39+Sv39x4) コールド2Dウォーク | (3+1)(3+1)−1 = **15 アクセス** |
| nested 定常残差（キャッシュ込み） | **≈ 2 アクセス / 8 ページ（単段の約2倍）** |

**核心**：レイテンシは縮められない、**重ねて隠す**だけ。必要並列度＝レイテンシ÷到着間隔（リトルの法則）。実装は「N個の複製ウォーカー」「1エンジン+Nコンテキスト」のどちらでも等価で、保存量は **同時outstanding本数**。

## 3. 設計の2大方針

ワイヤレート維持＝「41ns に1変換を完成」。2軸は補完的で、**A が W（実効レイテンシ）を下げ、B の必要量を直接減らす**（N ≈ W ÷ 到着間隔）。順序は「A→残差をB」。

### 軸A：1変換あたりメモリアクセスを減らす（W↓・帯域↓）
手段：PWC（上位段短絡）／64B leaf コアレッシング／DDT$・PDT$（文脈解決を消す）／スーパーページ／G-stage PWC＋S2 leaf コアレッシング／文脈タグ。IOTLB は streaming では再利用が無いので小さく。
- 判断：どのキャッシュ・容量・連想度、キャッシュ構造（direct/set-assoc/full）、コアレッシング幅、スーパーページ対応の有無。
- 考慮：効果は局所性前提に依存（崩れたら床へ）／メモリ帯域（accesses/page × wire rate）／キャッシュ汚染回避（構造分離）／面積・電力・タイミング（sky130）。
- sim 対応：`pwc_assoc, iotlb_assoc, coalesce`。

### 軸B：並列性／先読みで遅延を隠す（Wは不変, 重ねる）
手段：並列ウォーカーN／パイプライン化エンジン／prefetch（next-line, stride, RPT, DCPT, SMS）／深いバッファ＋MSHR／タグによる順序ずれ対応。
- 判断：N = ⌈W÷到着間隔⌉（A後のWで、+50〜100%マージン）、実装方式、バッファ深さ、prefetch距離・信頼度throttle。
- 考慮：メモリ outstanding 上限（N の天井）／MSHR 必須（コアレッシング効果）／順序ずれ対応／prefetch は“賭け”（自己無効化）／fault と投機の相互作用／front-end が 1ルックアップ/cycle。
- sim 対応：`num_walkers, buffer_size, prefetcher`。

## 4. アドレス変換の構造：DDTW / PDTW / PTW

処理順は **DDTW → PDTW → PTW(S1→S2)**。前2つは「文脈解決」、PTW は「アドレス解決」。

| | 解決対象 | 入力キー | ミス時 | キャッシュ | 単一文脈前提下 | サイジング駆動 |
|---|---|---|---|---|---|---|
| **DDTW** | デバイス文脈 | device_id | ≤3 | DDT$ | ほぼ常時ヒット(≈0) | 同時アクティブ device 数 |
| **PDTW** | プロセス文脈 | PASID | ≤3 | PDT$ | ほぼ常時ヒット(≈0)／PASID未使用なら省略 | 同時アクティブ PASID 数 |
| **PTW** | IOVA→GPA→SPA | IOVA(vpn)/GPA | 単段3/ネスト15/定常~2 | IOTLB+PWC(+G-stage PWC) | ★ワイヤレートの主駆動 | PTW レイテンシ |

- **DDTW/PDTW = 文脈解決**：単一文脈前提で定常ゼロコスト。設計は「容量（アクティブ文脈数）＋文脈タグ（switch時フラッシュ回避）＋構造分離」＝**容量問題**。ワイヤレートを駆動しない。
- **PTW = アドレス解決**：唯一のワイヤレート駆動＝**スループット問題**。必要N・バッファはここから。
- **MSI**：機能・隔離のため含める（特に仮想化）が、割り込みは3〜4桁低頻度＆固定集合でほぼ常時ヒット → 性能サイジングからは除外可。

### Sv39 / Sv39x4 とネスト
- Sv39：39bit VA、3段、各表512エントリ=4KiB、PTE=8B。
- Sv39x4：G-stage(2段目)用。GPA 41bit、root のみ 11bit=2048エントリ=16KiB、L1/L0 は4KiB。「入口だけ太い Sv39」。
- ネスト：IOVA→(VS,Sv39)→GPA→(G,Sv39x4)→SPA。guest PTE は GPA 空間にあり、読む前に毎回 G-stage で SPA 化。コールド 2D ウォーク = 4+4+4+3 = **15**。定常はキャッシュで ~2/8page。

## 5. 局所性の整理（最重要）

- **位置 vs 内容**：IOVA 連続 → leaf PTE が表内で連続（**位置**）→ 64B コアレッシング。データ物理連続（**内容**）→ スーパーページ可否。**両者は独立**。コアレッシングは内容に依存せず効く（頑健）。
- **GPA の二層局所性**（ネスト時、G-stage が変換する GPA）：
  - **表 GPA**（guest root/L1/leaf 表の所在）：leaf 表2MB / L1 1GB / root 512GB ごとしか動かない → **少数・長区間再利用＝時間局所性 極高**（値が散っても小ワーキングセット）。
  - **データ GPA**：guest が連続割当てなら **連続＝空間局所性**（S2 leaf コアレッシング可）。
  - どちらも **S1 PTE の中身(PPN)** が源泉。
- **G-stage が見るのは入力(GPA)**：SPA の分散は翻訳キャッシュ効率に無関係（スーパーページとDRAM行局所性にのみ効く）。
- **GPA = IOVA + 定数**：1つの連続・順方向マッピング領域の中だけ成立（+4kB IOVA=+4kB GPA, ページ番号ロックステップ）。崩れるのは：スキャッタギャザー/チャンク境界（C が片ごとに変化）、非単調マップ、リングラップ、穴/ガード、ページサイズ混在。**“1つの連続領域内”ならロックステップ、境界でジャンプ**。これは S1 の話で、GPA→SPA(S2) は独立。
- **単一プロセス ≠ GPA 連続**：文脈数は「どのページテーブルか」、GPA 連続はバッファ配置の話で独立。`malloc` が仮想連続でも物理が散るのと同じく、S1 ページテーブルは連続IOVAを散在GPAへ写せる。ジャンプを防ぐのは **IOVA≡GPA（恒等S1）** か **連続(hugepage)バッファ＋線形S1**。
- **G-stage PWC が効かなくなる**：GPA フットプリントが広く分散（distinct S2 L1 が容量超）／多文脈で和集合が大／G-stage 無効化頻発（live migration, balloon, page migration）／ホストが4kB裏打ち＋広域。なお S2 root/L2 はほぼ常に残るので **S2 L1 から先に失う＝なだらか**。

## 6. メモリ / AXI / DRAM とデータバッファ

- **DRAM 構造**：1T1C セルの行×列アレイ。ワード線で行選択→センスアンプ(row buffer)へ展開(ACTIVATE, tRCD)→列選択(CAS, tCL)→64Bバースト。**100ns は「行を開く代償」で行ごとに1回**。開いた行内の連続アクセスは CAS のみで安い。
- **64B 読み**：DRAM バースト(BL8×64bit=64B)＝キャッシュライン。レイテンシは固定なので 8B も 64B もほぼ同じ100ns → **コアレッシングはほぼタダ**。「64Bごとに100ns」ではなく「行ごとに100ns」。128B は同行なら 1 ACTIVATE + 複数CASで、2回目以降は安い。
- **AXI スプリットトランザクション**：AR(アドレス+タグ)→R(データ+タグ) が分離。アドレスを止めず連発でき、**同時 outstanding 本数＝並列ウォーク数の上限**。返りはタグで対応づけ。リード/ライトはチャネル別だが同じ DRAM 帯域を共有。
- **データバッファとI/Oブリッジ**：**4kB データ本体は I/O ブリッジが保持**（IOMMU は制御状態のみ）。3d「I/Oブリッジの最小バッファ」＝残差レイテンシの BDP（=100GB/s × W）。即時応答（ヒット/事前翻訳）ならデータ素通り、遅延応答（デマンドウォーク）なら N×4kB 保持。
- **ATS（事前翻訳）**：デバイスが将来アドレスを事前翻訳しデバイス側 ATC に保持→「翻訳済み」で書き込み→ブリッジのデータバッファが ATC エントリへ縮む。**プロトコルレベルの prefetch**。毎回ブロッキングで pull するのは往復増＝逆効果（バッファは保存量で移動するだけ）。

## 7. 前提パイプライン（フローチャートの6軸）

構造を変える軸を上に、規模を変える軸を下に。各軸でベースライン（採用）と代替（=感度）を持つ。

| 軸 | ベースライン（採用） | 代替（感度） | 駆動する判断 |
|---|---|---|---|
| 0 固定要件 | wire 100GB/s, 到着40.96ns, mem100ns | — | N・バッファのベースライン |
| 1 仮想化/段数 | **ネスト2段（採用）** | ベアメタル単段 | G-stage PWC, ~2x, MSI |
| 2 IOVAパターン | 連続/単調 | ランダム/非単調【崖】 | コアレッシング・PWC・prefetch・N |
| 3 GPA配置/S1 | 連続(hugepage)/線形S1 | 断片化/散るS1【なだらか】 | S2 コアレッシング, GPA=IOVA+C |
| 4 文脈多重度 | 単一 device/process | 複数/スイッチ【容量超で崖】 | DDT$/PDT$ 容量・文脈タグ・PDTW有無 |
| 5 DMAライフサイクル | ピン留め/登録 | streaming map/unmap【崖】 | invalidation 処理, fault |
| 6 メモリ outstanding | 十分 | 上限あり | N の天井, バンク並列 |
| レンズ | 定常 | 過渡(cold-start) | サイズは定常で、過渡は上流/ウォームアップ |

### SW/プラットフォーム能力前提（最上段：軸の枝を有効化/枝刈り）
| 機能 | 有効化 | 無い場合 |
|---|---|---|
| IOMMUスーパーページ＋guest/host hugepage | 変換激減・GPA/SPA連続・表極小 | 4kB粒度・キャッシュ大 |
| ATS(+ATC) | 事前翻訳・ブリッジバッファ→ATC | 全翻訳がホットパス・4kB×N保持 |
| PRI | デマンドページング | ピン留め必須(fault≈0) |

| lazy/batched＋選択的無効化 | 無効化率↓ | strict→per-page【崖】 |

| IOMMUによる連続IOVA再マップ | 物理が散っても連続IOVA | 連続IOVA前提が崩れる |

## 9. 3c と 3d の違い

- **3c**：バッファ無限を仮定 → 必要 **並列ウォーカー数 N**（=同時進行ウォーク数）。sim: `buffer_size=None` で `peak_walks`。
- **3d**：ウォーカー十分を仮定 → 必要 **最小バッファ B**（=滞在中の全トランザクション数。ヒット・MSHRコアレッシング相乗り・データ待ちを含む）。sim: `num_walkers=None` で `peak_buffer`。
- **別物**：ウォーカーは「進行中ウォーク」、バッファは「滞在中の全要求」。1ウォークに複数要求が乗る（ヒット/コアレッシング）ため **B ≧ N**。例（sim B構成）：peak_walks=1 だが peak_buffer=8。
- no-cache では両方8で一致（偶然）。だから独立な下限として別々に求める。ウォーカー床を割るとバッファでは救えない（実例E：walker4/buffer4→13.3M/s）。

## 10. 設計原則

- **レイテンシは隠す（縮めない）**：concurrency = W/到着間隔。
- **free-win vs bet**：コアレッシング/PWC＝崩れても“速くならないだけ”（採用）。prefetch/極小キャッシュ/ぎりぎりN＝賭け（自己無効化・マージン）。
- **崖 vs なだらか**：最悪が「破綻」でなく「鈍化」になるよう。
- **共通ケース最適化＋正しいフォールバック**：性能だけ前提依存、正しさは非依存。逸脱は遅くなるだけ。
- **置換より構造分離**：単調アクセスでは置換アルゴリズムはほぼ無意味。レベル別キャッシュ分離で streaming leaf が上位を汚さないようにする。
- **定常で設計、過渡は別レンズ**：一度きりの起動は上流バッファ＋ウォームアップで吸う。繰り返す過渡（文脈スイッチ/無効化）は容量・選択的無効化・ウォームアップで潰す。

## 11. 汎用性の階層（どこまで“速く”対応するか）

| 階層 | 内容 | 方針 |
|---|---|---|
| T0 正しさ | 変換・保護・fault・無効化・単段/2段・任意IOVA/文脈 | **完全汎用**（常に正しい） |
| T1 設計対象 | 仕様のバルク連続streaming | **フル最適化** |
| T2 劣化包絡 | 複数ストリーム・中文脈数・断片化・ネスト・スーパーページ | 安価・頑健な機能でなだらか劣化 |
| T3 包絡外 | 小パケットランダム・strict per-page無効化・fault多発 | 正しいが遅い（ワイヤレート非対象） |

作り込む（T1+T2を安く）：PWC・64Bコアレッシング・構造分離・MSHR・スーパーページ対応・ネスト＋文脈タグ・信頼度prefetch・N/バッファのパラメータ化・バッチ/選択的無効化。
作り込まない（T3はフォールバック）：800Gワイヤレートに乗らない病的ケース。

## 12. sky130 とマージン

- **必要N・バッファはクロック非依存**（ns で決まる）。クロックのために水増し不要。
- リソースマージン：実測ピークの **+50〜100%**（メモリテール・実サイクル・バースト）。バッファは安く高めに。
- タイミング：合成Fmax目標 **1.3〜1.5×**（slowコーナー＋P&R劣化）、**使えるサイクル予算 ~70%**、危険ブロック（CAM/調停）は早期試し合成で校正。**小さい単サイクルキャッシュ＋パイプライン化**にアーキ段階でコミット。
- メモリ帯域：ウォーク帯域 ≤ ~50%。
- sky130 で 400MHz は **攻めた目標**（標準セル/オープンフローの快適域 ~100〜250MHz）。PCIe Gen6/800GbE PHY は 130nm 不可だが **スコープ外**（本タスクは IOMMU デジタルブロック）。プロトタイプには可。

## 13. 面積・電力推定（合成前）

- メモリ支配なので **CACTI/OpenRAM(sky130) ＋ シミュレータの活動カウント** で相対比較が ±20〜50% で実用。
- 動的電力＝活動ベース（アクセス回数×1アクセスエネルギー＋FFクロック）、静的＝bit/gate×リーク。
- 推定 → 凍結 → 合成 → 誤差分解 → 校正係数、の順（推定を先に確定）。
- 配線・クロックツリーは合成前推定の構造的な穴 → 相対比較は信頼、絶対はP&Rまで。
- estimator はバッファの4kBデータを計上しない（I/Oブリッジ持ち）。制御bitのみ。

## 15. ネスト時キャッシュ構成（確定）

per-IOVA で S2 変換が要る GPA は4本、変化頻度の階層:

| S2変換の対象GPA | 出どころ | 変化頻度 | 扱い |
|---|---|---|---|
| root表 GPA | vsatp | 不変 | レジスタ化（ミスゼロ） |
| L1表 GPA | gL2 PTE | 1GB毎 | 極小・ほぼ常時ヒット |
| leaf表 GPA | gL1 PTE | 2MB毎 | 小・2MB毎ミス |
| データ GPA | gL0 PTE | 4kB毎 | 8枚コアレッシング |

キャッシュ集合（容量は出発点、sim で確定）:

| キャッシュ | レベル | 容量 | 連想 | タグ |
|---|---|---|---|---|
| IOTLB（結合 IOVA→SPA） | leaf結合 | 32–64 | 4-way | VPN＋device_id＋PASID＋VMID |
| S1 PWC | gL2/gL1 | 8–16 | full | VPN prefix＋level＋ctx |
| S2 PWC | S2 root/L1 | 8–16 | full | GPA prefix＋level＋VMID |
| 表GPA→SPA | S2 leaf(表) | 8–16 | full | GPA page＋VMID |
| データGPA→SPA | S2 leaf(data) | 32–64 | 4-way | GPA page＋VMID（**on/off**） |
| DDT$/PDT$/MSI$ | 文脈/割込 | =文脈数 / 8–16 | full | device_id / PASID / MSI |

要点:
- **root はレジスタ化**（単一文脈で不変・ミスゼロ）。
- **PWCは全非leafレベルを持つ**が合計は数エントリ（1エントリで2MB/1GB/512GBをカバーし前進スライド）。最深の非leaf(L1)が最高価値、上位は幾何級数的に価値減だが極小コストで持つ。
- **結合IOTLBがあれば独立データGPA→SPAは冗長**（S2データ結果は充填時にIOTLBへ畳み込み）。ただし**invalidation時はステージ分離が有効**（S1無効化でもS2結果温存）→ **on/offトグル**化。
- **表GPA→SPA と データGPA→SPA は必ず分離**（高churnのデータが少数・常時ヒットの表を汚染しない）。
- on-demand時の**ミスペナルティ(cycle)は種別別**に出す（IOTLBヒット/MSHR相乗り/PWC全ヒット/部分/フルコールド）。

## 16. キャッシュ実装メモ（連想度・ルックアップ・フラッシュ・SRAM）

- **連想度**：direct(1) / set-assoc(N) / full。キー=[tag｜index｜offset]。極小→**full(CAM、競合ゼロ・必ずヒット)**、大きめ→**4-way**。置換が要るのは assoc>1 のみ。
- **ルックアップ**：sequential（IOTLB先→ミス時PWC）/ parallel（全引き＋**最完成度ヒット優先エンコーダ** IOTLB>PWC-L1>PWC-L2>root）/ hybrid（IOTLB先→ミス時PWC並列）。完成度の階層で「最も深いヒットで打ち切り」。プリフェッチ全ヒット時はIOTLBが優先で勝つ。
- **フラッシュ**：flush-all（全valクリア＝コールド化）/ selective（address・range／**contextタグで文脈単位**）/ **generation方式（世代+1でO(1)一括、文脈別世代も可）**/ invalidation-queue（**batch/lazy既定**、strictは崖）/ **ATS伝播**（device ATCへ）。
- **SRAM最小容量**：極小キャッシュは**FF/レジスタCAM**で作る（SRAM周辺が無駄、フル連想に適）。複数レベルを**1 SRAMに同居は partitioning で可**（周辺償却）だが汚染トレードオフ。実機は FF-CAM(極小)＋小型SRAM(leaf/data) の混在。
- **汚染回避**：**物理分離が既定**（streaming leafが上位/文脈を蹴り出さない）。統合する場合は **way-partition＋scan-resistant置換＋厚タグ**。厚タグは**選択的無効化**にも有用。

## 17. ワーキングセット・頑健性・データ経路（補足）

- **容量は“総数”でなく“ワーキングセット”で決まる**：streamingは各レベルとも小・有界・前進スライド → **無限ストリームでも固定小容量**で足りる。「全レベルキャッシュ＝無限」ではない（rootはレジスタ1個）。
- **実転送長**：1転送＝MB〜GB（µs〜ms）で有界、境界は稀で償却。連続前提は転送内で成立。
- **ストライド適応（崖回避）**：**+4kBを焼き込まない**。領域ベースPWC・**ストライド検出prefetch（信頼度throttle）**・ライン取得coalescing → 一定ストライドが変わっても**なだらか劣化**、崖は**ランダムのみ**（フォールバックで正しく遅く）。特化先は「値+4kB」でなく「**WSが小さくストライド検出可能な連続流**」。
- **データ経路の2ケース（§6補足）**：① 即時応答（ヒット/事前翻訳）→ **データ素通り・保持最小**／② 遅延応答（デマンドウォーク）→ **データを N×4kB 保持**（=3d、I/Oブリッジが持つ）。**ブロッキングpull（翻訳完了後にデータ要求）は往復増で逆効果**（バッファは保存量で移動するだけ・ヒットで無駄）。正解は **ATS＝プロトコルprefetch**で②→①に動かし素通り化（飛行中状態を「4kBデータ」から「小さなATCエントリ」へ）。
