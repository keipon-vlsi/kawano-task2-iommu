# IOMMU マイクロアーキテクチャ — 最小成果物レポート (3a–3d)

Pre-training Program 2 (Tenstorrent / ADIP) — Task 2
対象: 800GbE NIC のワイヤレート DMA アドレス変換を支える IOMMU

本レポートは課題 (`doc/tt_pretraining_task2.md`) が要求する 2 つの成果物
**(1) 性能要件と提案マイクロアーキテクチャのレポート** および
**(2) シミュレータと結果** を提供する。採点対象の 3a–3d は §3–§6、提案アーキ

イベント駆動シミュレータ (`iommu_sim/`) の両方で導出し、両者の一致を示す。

## 2. 性能要件の導出

### 2.1 実効ワイヤレート
Ethernet は 8b/B のペイロード換算で **800 Gbps = 100 GB/s** が実効データレート。
PCIe Gen6 x16 = 128 GB/s はこれを上回るため**ボトルネックは Ethernet 側**であり、
IOMMU が支えるべき変換スループットは 100 GB/s 基準で決まる。

### 2.2 変換レート / 到着間隔 / サイクル予算
- 変換レート = 100×10⁹ / 4096 = **24.41 M translations/s**
- 到着間隔 (inter-arrival) = 4096 / 100×10⁹ = **40.96 ns ≈ 16.38 cycles** @ 400 MHz
- **サイクル予算**: 1 変換あたり平均 16.38 cycles 以内に処理を「開始」できれば
  ワイヤレートを維持できる (レイテンシ自体は隠蔽してよい。§2.3)。

### 2.3 設計原理: 「レイテンシは削減ではなく隠蔽する」
1 アクセス 100 ns のメモリレイテンシは到着間隔 40.96 ns より長い。よって
**直列処理ではワイヤレートに追いつけない**。Little の法則で必要な並列度を求める:

```
必要並列度 N = レイテンシ L / 到着間隔 Ta
```

これを N 個の並列ウォーカ、または N 個のアウトスタンディングなタグ付きメモリ
リードを持つ 1 本のパイプラインで実装する。「ウォーカ」は ALU ではなく
**小さな状態箱 + 未完了メモリリードを 1 つ保持する権利**である。

## 3. 3a — キャッシュなしのアドレス変換レイテンシ

### 解析
Sv39 は 3 段 (root → L1 → leaf)。キャッシュが無ければ各段で 1 メモリアクセス:

```
L_nocache = 3 levels × 100 ns = 300 ns
```

到着間隔 40.96 ns に対し約 7.3 倍。直列では 24.41 M/s に対し 3.33 M/s しか出ない。

### シミュレータ確認 (シナリオ A, 8000 連続リクエスト)
| 指標 | 値 |
|---|---|
| メモリアクセス / page | **3.000** |
| 平均レイテンシ | **300.0 ns** (p99 300.0 ns) |
| peak walks (必要 N) | 8 |
| peak buffer | 8 |
| 達成スループット | 24.39 M/s (無限資源なら充足) |

解析の 300 ns とシミュレータが完全一致。Little の法則の逆算
`N = 300 / 40.96 = 7.32 → 8` も peak walks と一致する。
→ 図 `figures/fig1_latency_vs_config.png` の最左バー。

## 4. 3b — キャッシュ構造 (IOATC) ありのレイテンシ

IOATC を**レベルごとに分離**して配置する: IOTLB (最終 leaf)、PWC (上位 PTE)、
DDT$ / PDT$ (デバイス/プロセスコンテキスト)、MSI$ (割り込み変換)。

### 4.1 PWC + 64 B leaf coalescing (シナリオ B)
- **PWC**: 上位 2 段 (L1=1 GB, root=512 GB 単位) は長区間で再利用されるため、
  ウォームアップ後はほぼ常にヒット。上位段のメモリアクセスが消える。
- **64 B coalescing (最大の安価な勝ち筋)**: DRAM は 64 B バースト = **8 PTE** を
  1 アクセスで返す。連続 IOVA では leaf PTE も連続するため、**8 ページあたり 1
  アクセス**に削減。これは PTE の配置連続性 (= IOVA 連続) のみに依存し、データの
  物理配置とは独立。

| 指標 (シナリオ B) | 値 |
|---|---|
| メモリアクセス / page | **0.127** (≈ 1/8 + 上位ミス残差) |
| 平均レイテンシ | **24.6 ns** (p99 100.0 ns) |
| peak walks (必要 N) | **1** |
| IOTLB hit / MSHR coalesced / true walk | 4965 / 2035 / 1000 |

300 ns → 24.6 ns へ **約 12 倍**短縮。mem/page は 3.0 → 0.127 へ **約 24 倍**削減。

### 4.2 + プリフェッチ (シナリオ C, next-line)
信頼度スロットル付き next-line プリフェッチで leaf line を先読みし、需要側が
ヒットになる。

| 指標 (シナリオ C) | 値 |
|---|---|
| メモリアクセス / page | 0.128 |
| 平均レイテンシ | **2.7 ns** (p99 2.5 ns = 1 cycle) |
| peak walks (必要 N) | 3 |

需要パスの観測レイテンシはほぼ 1 cycle に潰れる。→ 図 `fig1`, `fig3`。

### 4.3 DDT$ / PDT$ / MSI$ — 定常状態の寄与は ~0
単一連続ストリームでは device_id / process_id は**一定**。したがって DDT / PDT /
MSI の各コンテキストテーブルはストリーム開始時に**一度だけ cold で歩かれ**、以降は
各キャッシュからヒットする。シミュレータに `ContextCachedCost` (DDT 3 段 + PDT 2 段
を初回のみ課金) を追加して定量化した:

| 構成 | mem/page | cold コンテキストアクセス | 償却レイテンシ |
|---|---|---|---|
| B (コンテキスト費用なし) | 0.1271 | 0 | 0 ns/page |
| B + DDT$/PDT$ (初回 cold) | 0.1280 | +7 (≈5 段 + coalescing 揺らぎ) | **+0.088 ns/page** |

→ 8000 ページ全体で 1 回限りの cold コストが償却され、**定常状態の付加レイテンシ
は実質 0**。MSI$ も同様 (割り込みベクタは少数・高再利用)。
データ: `context_cache.csv`。

**まとめ (3b)**: レベル分離 IOATC + PWC + 64 B coalescing で
**300 ns → 約 25 ns (定常)、プリフェッチ併用で ~2.7 ns**。DDT$/PDT$/MSI$ は
ウォーム後 ~0。

## 5. 3c — ワイヤレート維持に必要な並列ページテーブルウォーク数 (無限バッファ前提)

### 解析 (Little の法則)
```
N = L / Ta
```
- キャッシュなし: N = 300 ns / 40.96 ns = 7.32 → **8**
- キャッシュあり (定常 B): 残差 mem/page ≈ 0.127、実効レイテンシ ≈ 24.6 ns →
  N = 24.6 / 40.96 ≈ 0.6 → 定常では **1** で足りる
- プリフェッチ併用 (C): cold line 先読みのバースト性で瞬間 peak が上がり **3**

### シミュレータ確認 (無限資源 `num_walkers=None, buffer_size=None`)
| シナリオ | peak walks (= 必要 N) |
|---|---|
| A no-cache | **8** |
| B PWC+coalescing | **1** |
| C +prefetch | **3** |
| D random IOVA (最悪) | **8** |

### ウォーカ数スイープ — ワイヤレートの「崖」
no-cache 構成で無限バッファのままウォーカ数を 1→12 に振ると、スループットは
N に比例して直線的に増加し **N=8 でワイヤレート (24.41 M/s) に到達して飽和**する。
N<8 では崖の下に落ちる (N=4 → 13.3 M/s, シナリオ E)。
→ 図 `figures/fig2_throughput_vs_walkers.png`、データ `walker_sweep.csv`。

| N | 達成スループット (M/s) | wire 充足 |
|---|---|---|
| 4 | 13.33 | ✗ |
| 7 | 23.32 | ✗ |
| **8** | **24.39** | ✓ |
| 12 | 24.39 | ✓ |

**結論 (3c)**: 一般フォールバック (キャッシュなし / 非単調 IOVA) を安全に支える
**最悪ケースの必要 N = 8**。CLAUDE.md のマージン指針 (+50–100%) に従い、
**12–16 個の並列ウォーカ (= MSHR/アウトスタンディングリード)** を実装することを推奨。
定常のキャッシュ最適パスでは N=1–3 しか使わないが、安価な状態箱なので過剰に持つ
コストは小さく、ロバスト性 (崖回避) の見返りが大きい。

## 6. 3d — ワイヤレート維持に必要な最小バッファ (十分なウォーカ前提)

I/O ブリッジの transaction buffer は**処理中 (in-flight) の変換要求**を保持する。
十分なウォーカがある前提では、バッファ占有 = 同時 in-flight 数 ≈ Little の法則の N。

### 解析
```
buffer ≈ L / Ta = 300 / 40.96 = 7.32 → 8 (no-cache 最悪)
```

### シミュレータ確認
- **peak buffer** は全シナリオで **8** (A/B/C/D いずれも 8)。
  キャッシュ最適時でも、ストリーム開始直後の cold-start で 8 段が一時的に埋まる
  (定常では 1–3 まで下がるが、**cold-start transient が最小バッファを決める**)。
- バッファ深度スイープ (no-cache, ウォーカ十分): 深度 8 で初めてワイヤレート充足、
  7 以下では崖。→ 図 `figures/fig4_throughput_vs_buffer.png`、`buffer_sweep.csv`。

| buffer | 達成スループット (M/s) | wire 充足 |
|---|---|---|
| 4 | 13.33 | ✗ |
| 7 | 23.32 | ✗ |
| **8** | **24.39** | ✓ |

**結論 (3d)**: 定常 vs cold-start を区別すると、定常占有は 1–3 だが
**cold-start transient のため最小 8 段**が必要。マージン (+50–100%) を加えて
**16 段の transaction buffer** を推奨。これによりバースト・コンテキスト切替・
一時的な非単調アクセスにも崖なく対応できる。

## 7. 提案 IOMMU マイクロアーキテクチャ

### 7.1 ブロック図

```
          IOVA (device DMA addr)
              │
   ┌──────────▼───────────────────────────────────────────┐
   │  I/O Bridge  —  Transaction Buffer (depth 16)         │  ← 3d: 最小8+マージン
   │   (in-flight 変換要求, back-pressure で arrival 制御)  │
   └──────────┬───────────────────────────────────────────┘
              │ admit
   ┌──────────▼───────────┐   分離型 IOATC (レベルごとに独立)
   │  Translation Pipeline │   ┌────────────┬───────────────┐
   │  + Prefetch (next-    │──▶│ IOTLB(leaf)│ PWC (upper PTE)│
   │    line, conf-throttle)│  ├────────────┼───────────────┤
   │                       │   │  DDT$      │ PDT$ │ MSI$    │
   └──────────┬───────────┘   └────────────┴───────────────┘
              │ miss → walk         (DDT$/PDT$/MSI$ は単一ctxで
   ┌──────────▼────────────────────────┐   ウォーム後ヒット)
   │  Walker Pool  (12–16 walkers)      │  ← 3c: 最悪8+マージン
   │  = N 個のアウトスタンディング       │
   │    タグ付きメモリリード             │
   │  + MSHR (同一64Bラインのミスを集約) │  ← 64B coalescing の要
   └──────────┬────────────────────────┘
              │ 64 B burst read (= 8 PTE)
   ┌──────────▼───────────┐
   │  AXI / Main Memory    │  100 ns latency
   └──────────────────────┘
```

### 7.2 設計判断と根拠 (CLAUDE.md の確立原理を適用)

| 要素 | 選択 | 根拠 |
|---|---|---|
| **64 B leaf coalescing + MSHR** | 必須・最優先 | 「最大の安価な勝ち筋」。1 アクセスで 8 PTE。mem/page を 3.0→0.13 に。**free win**。 |
| **PWC (上位 PTE 専用)** | 必須 | 上位段は長区間再利用。leaf ストリームから分離して常時ヒット維持。 |
| **レベル分離 IOATC** | IOTLB/PWC/DDT$/PDT$/MSI$ を独立に | 置換ポリシは単調 IOVA でほぼ無意味 (再利用なし)。本質的レバーは**構造的分離**で、ストリーミング leaf トラフィックがホットな上位エントリを追い出さないこと。 |
| **leaf IOTLB** | 小容量で十分 | ストリーミングでは leaf の再利用がほぼ無い。面積は PWC + coalescing に割く。 |
| **Walker pool = 12–16** | 最悪 N=8 + マージン | レイテンシは隠蔽。一般フォールバック (no-cache / random IOVA) を崖なく支える。 |
| **Transaction buffer = 16** | 最悪 8 + マージン | cold-start transient が下限を決める。バースト/切替に余裕。 |
| **Prefetch (next-line, 信頼度スロットル)** | オプション (bet) | 連続 IOVA で観測レイテンシを ~1 cycle に。**自己無効化**設計で、外れたら勝手に止まり一般経路にフォールバック。 |

### 7.3 ロバスト性の方針
- **free win を優先、bet は自己無効化**: coalescing と PWC (確実な利得) を土台に、
  prefetch や小型コンテキストキャッシュ (賭け) は信頼度スロットルで外れても害が
  ないようにする。
- **崖ではなく緩やかな劣化**: 非単調 IOVA やコンテキスト切替で最適化が崩れても、
  ウォーカ 12–16 / バッファ 16 が no-cache 相当の最悪ケースを吸収するため、
  性能は「遅くなる」だけで「壊れない」(シナリオ D は 24.40 M/s を維持)。

## 9. 前提と設計上の判断 (Assumptions)

- **変換方式**: 課題は変換方式を明示しないため、RISC-V IOMMU の Sv39 (39-bit,
  3-level) を単段変換の基準とした。3a の 3×100 ns はこの 3 段に対応。ネスト
  (Sv39 + Sv39x4, 2D walk) は最小成果物の範囲外とし、コストモデル `NestedCost`
  として拡張点のみ用意した。
- **メモリレイテンシ**: 100 ns/アクセスを固定とし、行アクティベーション・バンク
  競合・帯域飽和は最小モデルでは無視 (64 B バースト = 8 PTE のみモデル化)。
- **到着パターン**: 主結果は連続 4 kB ページ (単調 IOVA)。非単調・ランダムは
  感度解析 (シナリオ D) として扱う。
- **DDT$/PDT$/MSI$**: 単一連続ストリームで device_id/process_id 一定と仮定。
  初回 cold walk のみ課金し定常 ~0 を示した (`ContextCachedCost`)。マルチコンテキスト
  切替時の感度はフェーズ 3 の課題。
- **ヒット完了 = 1 cycle (2.5 ns)**: プリフェッチ命中時の観測レイテンシ下限として
  採用 (CLAUDE.md / ASSUMPTIONS.md と整合)。
- **シム ≠ タイミング**: 本シミュレータは「アーキテクチャが十分な並列度を持つ」
  ことを示すもので、合成後に 400 MHz が閉じることは保証しない。実装では
  Fmax ≈ 1.3–1.5× 仕様、使用サイクル予算 ~70%、N/buffer/MSHR を +50–100% で
  プロビジョニングする (本レポートの 12–16 / 16 はこの方針に従う)。
