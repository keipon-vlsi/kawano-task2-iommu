# マルチレベル PWC ルックアップ：並列+優先 vs 直列（リーフ近い順）

VS PWC を L1（リーフに近い, tag=VPN[26:9] 18b）/ L2（tag=VPN[26:18] 9b）/ root（レジスタ）
の 3 レベルで持つとき、**どう引くか**の比較。キャッシュ構造そのものとは別軸。
スクリプト `syn/pwc_level_compare.py`、モジュール `pwc/pwc_lvl_par.sv` / `pwc/pwc_lvl_seq.sv`。

- **parallel（並列+最リーフ優先, 現行 iommu_top 方式）**：3 レベルを毎アクセス同時にルックアップし、
  最もリーフに近いヒットを優先 mux で選ぶ（near>mid>root）。**1 サイクル**。
- **sequential（直列, リーフ近い順）**：near をまず引き、ミスしたら次サイクルで mid、さらにミスなら
  root。小さな FSM で **1 レベル/サイクル**。**1〜3 サイクル**（near がほぼ常にヒットする steady state
  では ~1）。

## 測定（同一フロー hd / 機能 PASS / post-opt @1.0ns target, ideal clock）

| 方式 | func | Fmax [MHz] | area [µm²] | cells | DFF | logic depth | レイテンシ |
|---|---|---|---|---|---|---|---|
| parallel + priority | ✅ | **416.8** | **10,167** | **775** | **248** | 7 | **1 cycle（常に）** |
| sequential（leaf-first） | ✅ | 375.6 | 12,508 | 923 | 303 | 7 | 1–3 cyc（steady ~1）|
| **Δ seq vs par** | | **−9.9%** | **+23.0%** | +19% | +22% | 同 | +レイテンシ |

## 解釈

- **本構成では parallel が全 PPA 軸で勝ち**：Fmax +9.9%・面積 −23%・DFF −22%・レイテンシ 1 cyc 固定。
- **なぜ sequential が不利か**：
  - 3 レベルとも tag 幅・内容が違うので**比較器を共有できない**＝直列にしても比較器の数は減らない
    （storage も同じ）。一方で **FSM（状態・VPN ホールド・resp_valid・レベルラッチ）の制御オーバーヘッド**が
    純増（DFF 248→303, +55）。
  - 「1 レベル/サイクルで論理が浅くなる→ Fmax 向上」を期待したが、**実測の論理段数は同じ 7**。並列の
    優先 mux（near>mid>root の 2 段 2:1）は浅く、3 レベルだけなら段数差を生まない。逆に FSM の次状態論理
    が乗り、Fmax はむしろ −10%。
  - **レイテンシは +1〜2 サイクル**（steady state の near ヒットなら +0 に近いが、それでも 1 サイクルは
    並列と同じで、ミス時だけ伸びる）。利点なし。
- **唯一 sequential が有利になり得るのは動的電力**：steady state で near が毎回ヒットするなら、mid/root の
  比較は「結果を使わない」ので、活性化を抑えれば毎アクセスの toggling を減らせる可能性。ただし本フローの
  flat-activity 見積もりではこの差は出ず、定量化には activity-aware（VCD）測定が必要。かつ FSM の常時稼働
  ぶんが相殺し得る。**面積/Fmax/レイテンシの確実な不利を上回るほどの電力利得は、レベル数が少ない本構成では
  考えにくい**。

## 結論

3 レベル PWC のように**レベル数が少なく各キャッシュが小さい**場合、**並列+最リーフ優先（現行方式）が
面積・Fmax・レイテンシすべてで優位**。直列（リーフ近い順）は比較器を共有できないため面積を減らせず、
FSM オーバーヘッドと追加レイテンシだけが乗る。直列が効くのは「レベル数が多く、1 個の大きな比較器を
レベル間で共有して面積を削れる」ような構成（本タスクには該当せず）。
→ 現行 iommu_top の **並列+most-complete-hit が妥当**。
